  • Right Reviewer for the Job
  • Be Diplomatic and No Finger-pointing

Effective Code Review Process (3)

  • Improve Code Using Tools Before PR
  • Keep Small: Max 600 LOC, max 15 files. Best: Under 200 LOC
  • Optimal Review Time: 15-30 mins. Keep under 1hr
  • One PR per Bug/Feature/Task
  • Generated Files Should be Separate

Query Expenses of an Employee/a Team

  1. Authentication
  2. Resource Availability
  3. Authorization
  4. Quota Management
  5. Obtain Data
  6. Report Errors

Take 1. Can You Spot Issues?

def getExpenseData(request):
    if isAuthenticated(request.user) \
        && targetExists(request.target) \
        && withinQuota(request.user) \
        && isAuthorized(request.user, request.target):
        result = queryData(request.target)
        return result.json()
    else:
        raise AccessDenied()

Take 1. Can You Spot Issues?

  1. The condition chain is complex
  2. AccessDenied is too vague, too little info
  3. Error reporting needs imporiving
  4. No logging! How to know when/where issues happen?

Take 2. Can You Spot Issues?

def getExpenseData(request):
    if isAuthenticated(request.user):
        if targetExists(request.target):
            if withinQuota(request.user):
                if isAuthorized(request.user, request.target):
                    result = queryData(request.target)
                    return result.json()
                else:
                    logger.error("The user is not authorized to access the target!")
                    raise AccessDenied()
            else:
                logger.error("Invalid target!")
                raise InvalidTarget()
        else:
            logger.error("Over quota for the user!")
            raise OverQuota()
    else:
        logger.error("Invalid user!")
        raise InvalidUser()

Take 2. Can You Spot Issues?

  1. Nesting is too deep!
  2. Code on each line gets too long
  3. Logging is not perfect, only marginally better
  4. The most important: Got a Bug!
  5. Switched else branch for InvalidTarget & OverQuota

Some Notes on Logging

  1. Make sure logging is sufficient
  2. Use logging level correctly: FATAL, ERROR, WARN, INFO, DEBUG, TRACE
  3. Logging must contain useful information
# BAD:
logger.error("The user is not authorized to access the target!")

# GOOD:
logger.error(
    "The user %s is not authorized to access the target %s" % (
        request.user, request.target))

Take 3. We Are Done!

def getExpenseData(request):
    if not isAuthenticated(request.user):
        logger.error("Invalid user: %s" % request.user)
        raise InvalidUser()
    if not targetExists(request.target):
        logger.error("Invalid target: %s" % request.target)
        raise InvalidTarget()
    if not withinQuota(request.user):
        logger.error("Over quota for the user: %s" % request.user)
        raise OverQuota()
    if not isAuthorized(request.user, request.target):
        logger.error("The user %s is not authorized to access the target %s" % (request.user, request.target))
        raise AccessDenied()
    result = queryData(request.target)
    return result.json()

Take 3. Commentary

  1. Practice defensive programming
  2. Return as soon as you can
  3. Avoid deep nesting
  4. Handle all errors & return values
  5. Provide sufficient logging level & contextual information
  6. We can split 5 steps above into 5 middleware layers

Buffer Overflow

Commentary

  1. Use static analysis to detect potential buffer overflow cases
  2. Always do boundary checking: == is not sufficient
  3. Do not use old unsafe functions like sprintf/vsprintf
  4. Prefer functions with boundary: snprintf/vsnprintf
  5. Practice defensive programming and don't trust user input!

Stack Overflow

How to cause a Stack Overflow

def fib(n):
    if n < 2:
        return 1
    else:
        return fib(n - 1) + fib(n - 2)

Stack Overflow

How to cause a Stack Overflow

class Record
{
    char array[1500000]; // Max size anticipated
};

void processData(int input, int anotherInput) // Simplified code
{
    Record rec;
    if (check(input)) {
        processData(input - 1, anotherInput);
    }
}

Stack Overflow

Commentary

  1. Design your algo carefully when using recursion
  2. Write tail-recursive functions if you can
  3. Use memoization/caching
  4. Use std::stack & loop to achieve the same effect as recursion but using heap
  5. Use heap (malloc/free/new/delete) if you need to allocate large memory

Integer Overflow: Google Binary Search

Adapted to fit this screen from source

1:     public static int binarySearch(int[] a, int key) {
2:         int low = 0;
3:         int high = a.length - 1;
4:         while (low <= high) {
5:             int mid = (low + high) / 2;
6:             int midVal = a[mid];
7:             if (midVal < key)       low = mid + 1
8:             else if (midVal > key)  high = mid - 1;
9:             else                    return mid; // key found
10:        }
11:        return -(low + 1);  // key not found.
12:    }

Integer Overflow: Google Binary Search

The issue is on the line 5


5:         int mid = (low + high) / 2;

Integer overflow when: (low + high) > 231 - 1

Integer Overflow: Google Binary Search

Solutions

5:         int mid = low + (high - low) / 2;

// The author also suggested a hacky way for C/C++
5:         int mid = ((unsigned int)low + (unsigned int)high) >> 1;

// Verdict: Use unsigned int in API and use the 1st approach

Integer Overflow: Google Binary Search

Key Takeaways

  1. Be aware of the numerical limits (integral or floating points)
  2. Write test cases that cover corner/edge cases
  3. Don't mix signed and unsigned integers
  4. Use linter/static analysis
  5. Get your code reviewed!

Lack of Abstraction: Memory Leak

Designing safe APIs is tricky

// Public API to access DB
class DbConn
{
public:
    // Constructors + Destructor
    ~DbConn() { /* Close connection if open */ }
    bool open(const std::string& dbURI);
    void close();
    // Open DB connection before executing
    int execute(const std::string& stmt, Result* out);
    bool isOpen() const;
    ...
};

Lack of Abstraction: Memory Leak

Practical Issues

  1. Clients use DbConn directly and have to open/close/execute explicitly
  2. Clients allocate DbConn on heap and forget to close
  3. Clients don't close DbConn on exception
  4. Memory is leaked due to clients' mistakes!
  5. DB server has too many file descriptors open!

Lack of Abstraction: Memory Leak

Lack of Abstraction: Memory Leak

Solution: Have Proper Abstraction

// Public API to access DB
class DbConnPool
{
public:
    static DbConnPool* createDbPool(const std::string& dbURI);
    ~DbConnPool(); // Take care of releasing resources
    // Fool-proof API, just execute
    int execute(const std::string& stmt, Result* out);
    ...
private:
    // Take care of constructors
};

Lack of Abstraction: Memory Leak

Commentary on DbConn(Pool)

  1. Design simple & fool-proof APIs. Don't trust users ;-)
  2. Some prefer a Singleton DbConnPool for all DBs
  3. Be careful in what to expose via public API
  4. Design review is so vital!

The problems (1/3)

  1. CloudFlare edge servers leaks some sensitive data: HTTP cookies, HTTP POST bodies, auth tokens, etc
  2. Search engines cache some data
  3. Memory is leaked on the CloudFlare servers

Verify Generated Code: CloudFlare & Ragel

The problems (2/3)

  1. CloudFlare parses HTML on the fly
  2. CloudFlare developed a new parser to replace Ragel and the issue occurred
  3. HTML parsing for email obfuscation, HTTP to HTTPS, server-side excludes, etc

The problems (3/3). Read on CloudFlare's blog

// CloudFlare blames Ragel's generated code as the root cause

if ( ++p == pe )

Verify Generated Code: CloudFlare & Ragel

Commentary

  1. Ragel fires back and points out that CloudFlare has been using the parser generater wrong
  2. Ragel will make code change to make generated code safer (don't trust users!)

Verify Generated Code: CloudFlare & Ragel

Solution

  1. Generated code must be tested and reviewed too
  2. Avoid using generated code blindly
  3. Prefer parser combinators to parser generators with write-only syntax & overly complex code
  4. Read the paper Writing parsers like it is 2017
