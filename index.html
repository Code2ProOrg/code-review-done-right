Effective Code Review Process (1/2)

  • Design & Technical Review Before Coding
  • Clear Context: BF/Feature/Refactor/Rewrite
  • Clear Scope: Who/What Part is Affected
  • Use Diagrams, Graphs and Charts for Clarity
  • Right Reviewer for the Job

Effective Code Review Process (2/2)

  • Keep Small: Max 1000 LOC, max 20 files
  • Give Timeline & Priority
  • One PR per Bug/Feature/Task
  • Be Diplomatic and No Finger-pointing
  • Improve Code Using Tools Before PR

Use a Code Formatter

  • C/C++: Clang-format, Astyle
  • Python: AutoPEP8, Google YAPF
  • JS: js-beautify, prettier
  • Go: gofmt
  • OCaml: ocp-indent
  • Astyle also claims to format C#, Java, Objective-C

Use a Linter/Static Analysis (C++)

  • Use g++/clang with special flags
  • CppCheck
  • Coverity - free for FOSS
  • Codacy - free for FOSS

Use a Linter/Static Analysis (Python)

  • Pylint
  • OpenStack Bandit security linter
  • Coverity - free for FOSS
  • Codacy - free for FOSS
  • pyflakes - faster but limited

Use a Linter/Static Analysis (JS)

  • ESLint - can add custom rules
  • JSHint - forked JSLint
  • JSLint - the oldest
