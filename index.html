Effective Code Review Process (1/3)

  • Design & Technical Review Before Coding
  • Clear Context: BF/Feature/Refactor/Rewrite
  • Clear Scope: Who/What Part is Affected
  • Have Check-list for Types of Projects

Effective Code Review Process (2/3)

  • Give Timeline & Priority
  • Use Diagrams, Graphs and Charts for Clarity
  • Right Reviewer for the Job
  • Be Diplomatic and No Finger-pointing
  • Any Uncaught Bug is Coder's Responsibility

Effective Code Review Process (3/3)

  • Improve Code Using Tools Before PR
  • Keep Small: Max 600 LOC, max 15 files
  • Optimal Review Time: 15-30 mins. Keep under 1hr
  • One PR per Bug/Feature/Task
  • Generated Files Should be Separate

Query Expenses of an Employee/a Team

  1. Authentication
  2. Resource Availability
  3. Authorization
  4. Quota Management
  5. Obtain Data
  6. Report Errors

Take 1. Can You Spot Issues?

def getExpenseData(request):
    if isAuthenticated(request.user) \
        && targetExists(request.target) \
        && withinQuota(request.user) \
        && isAuthorized(request.user, request.target):
        result = queryData(request.target)
        return result.json()
    else:
        raise AccessDenied()

Take 1. Can You Spot Issues?

  1. The condition chain is complex
  2. AccessDenied is too vague, too little info
  3. Error reporting needs imporiving
  4. No logging! How to know when/where issues happen?

Take 2. Can You Spot Issues?

def getExpenseData(request):
    if isAuthenticated(request.user):
        if targetExists(request.target):
            if withinQuota(request.user):
                if isAuthorized(request.user, request.target):
                    result = queryData(request.target)
                    return result.json()
                else:
                    logger.error("The user is not authorized to access the target!")
                    raise AccessDenied()
            else:
                logger.error("Invalid target!")
                raise InvalidTarget()
        else:
            logger.error("Over quota for the user!")
            raise OverQuota()
    else:
        logger.error("Invalid user!")
        raise InvalidUser()

Take 2. Can You Spot Issues?

  1. Nesting is too deep!
  2. Code on each line gets too long
  3. Logging is not perfect, only marginally better
  4. The most important: Got a Bug!
  5. Switched else branch for InvalidTarget & OverQuota

Some Notes on Logging

  1. Make sure logging is sufficient
  2. Use logging level correctly: FATAL, ERROR, WARN, INFO, DEBUG, TRACE
  3. Logging must contain useful information
# BAD:
logger.error("The user is not authorized to access the target!")

# GOOD:
logger.error(
    "The user %s is not authorized to access the target %s" % (
        request.user, request.target))

Take 3. We Are Done!

def getExpenseData(request):
    if not isAuthenticated(request.user):
        logger.error("Invalid user: %s" % request.user)
        raise InvalidUser()
    if not targetExists(request.target):
        logger.error("Invalid target: %s" % request.target)
        raise InvalidTarget()
    if not withinQuota(request.user):
        logger.error("Over quota for the user: %s" % request.user)
        raise OverQuota()
    if not isAuthorized(request.user, request.target):
        logger.error("The user %s is not authorized to access the target %s" % (request.user, request.target))
        raise AccessDenied()
    result = queryData(request.target)
    return result.json()

Take 3. Commentary

  1. Practice defensive programming
  2. Return as soon as you can
  3. Avoid deep nesting
  4. Handle all errors & return values
  5. Provide sufficient logging level & contextual information
  6. We can split 5 steps above into 5 middleware layers

Type Safety Matters

Take 1. What about the Python version?

def update_team_balance(team, balance):
    # Use team.ID
    ...

def update_user_balance(user, balance):
    # Use user.ID
    ...

# Duck-typing is a double-edge sword
# By accident - will update a team whose ID = user.ID
update_team_balance(user, 1000000)

It's not quite safe in Python yet :-(

Type Safety Matters

Take 2. Better & safer Python version

class User:
    def userId(self): return self.ID

class Team:
    def teamId(self): return self.ID

def update_team_balance(team, balance):
    # Use team.teamId()

def update_user_balance(user, balance):
    # Use user.userId()

update_team_balance(user, 1000000) # Won't work!

Type Safety Matters

Take 2. One can use distinct properties too

class User:
    def __init__(self, userId):
        self.userId = userId

class Team:
    def __init__(self, teamId):
        self.teamId = teamId

# This function requires teamId, make sure User doesn't have it
update_team_balance(user, 1000000)

Key Takeaways

  1. Have proper design review before coding
  2. Type safety is your best friend
  3. Use distinct types whenever you can
  4. Duck-typing is a two-edged sword: Use distinct names for methods and props across diff types
  5. Use type annotation & static analysis tools
  6. Consider type-safe languages

Adapted to fit this screen from source

1:     public static int binarySearch(int[] a, int key) {
2:         int low = 0;
3:         int high = a.length - 1;
4:         while (low <= high) {
5:             int mid = (low + high) / 2;
6:             int midVal = a[mid];
7:             if (midVal < key)       low = mid + 1
8:             else if (midVal > key)  high = mid - 1;
9:             else                    return mid; // key found
10:        }
11:        return -(low + 1);  // key not found.
12:    }

Integer Overflow: Google Binary Search

The issue is on the line 5


5:         int mid = (low + high) / 2;

Integer overflow when: (low + high) > 231 - 1

Integer Overflow: Google Binary Search

Solutions

5:         int mid = low + (high - low) / 2;

// The author also suggested a hacky way for C/C++
5:         int mid = ((unsigned int)low + (unsigned int)high) >> 1;

// Verdict: Use unsigned int in API and use the 1st approach
